Documentation GitHub

The crawl engine

Crawling is spider, vendored as a dependency. This package does not reimplement a crawler — it configures a spider::website::Website from the request dict (limit, depth, subdomains, robots, delay, concurrency, timeout, blacklist, proxies, user-agent) and calls scrape(), crawl(), or crawl_sitemap(). Results come back as { url, status, html? } per page.

One runtime, synchronous surface

spider's crawl methods are async and spawn tokio tasks internally, so they must run inside a tokio runtime. The cdylib owns a single multi-thread runtime, created lazily in a OnceCell and reused for the life of the process. Every network export drives spider with block_on, so the stryke side stays a plain synchronous JSON call — no async leaks across the FFI boundary.

The FFI bridge

Each scrape__* export is a JSON-string-in / JSON-string-out extern "C" function. stryke's bridge resolves the symbols listed in stryke.toml's [ffi] table at first use Scrape, passes a JSON args dict per call, and frees the returned string via stryke_free_cstring. Handlers run under catch_unwind, so a panic becomes an { "error": … } reply instead of unwinding across the boundary.

Pure extraction

The extract family runs html5ever-backed CSS selection (the scraper crate) over an HTML string with no network. Selector specs carry an optional trailing @attr to pull an attribute instead of text; extract_table maps a <table> to header-keyed records; extract_links resolves relative hrefs against a base via the url crate; structured parses JSON-LD script blocks and OpenGraph / Twitter meta tags. These are pure functions — unit-tested in the crate and exercised by the offline surface test with nothing to connect to.

Dependencies

spider (crawl engine, with sitemap), scraper (html5ever CSS selection), url (link resolution), tokio (the embedded runtime), serde / serde_json for the FFI payloads, anyhow + once_cell for error handling and the runtime cell.
